HLD — End-to-End System Architecture

At a high level, this is not just a document generator. It is a two-stage lineage system: a deterministic scanner creates structured evidence, then an LLM agent performs targeted investigation and writes reviewed lineage decisions.

LLD — Deterministic Engine Workflow

The scanner is the evidence layer. It does deterministic parsing, generates hashes, creates cache records, builds normalized content, extracts edges, and writes the files that the LLM uses for review.

01
Load config

load_config() reads analyzable extensions, ignore rules, evidence limits, and confidence thresholds.

02
Discover files

discover() separates analyzed files, inventoried files, and ignored files.

03
Hash source

file_sha256() calculates raw file hash for change detection and cache keying.

04
Cache lookup

cache_key() uses engine version + file hash + config + sqlglot availability.

05
Analyze file

analyze_file() parses notebooks, SQL, Python, YAML metadata, and PySpark calls.

06
Store content

store_content() writes normalized text to content_store/ and returns content records.

07
Extract lineage

extract_sql() and PySparkVisitor produce probable or partial edges.

08
Write outputs

write_outputs() writes JSON, CSV, markdown, change summary, and LLM manifest.

SQL path

SQL cell / spark.sql()→sqlglot.parse(..., read="databricks")→source tables→target tables→LineageEdge
  • Targets are detected from CREATE, INSERT, MERGE, and UPDATE.
  • All table references except target tables are treated as sources.
  • Complete source + target gives probable status with scanner confidence around 0.92.
  • Only source or only target becomes a partial edge for LLM review.

PySpark path

Python AST→track string variables→.table() reads→.saveAsTable() writes→edges/unresolved
  • Static string assignments are stored so simple dynamic table variables can be resolved.
  • Unresolved dynamic calls are written to unresolved_items.json.
  • Path-based .load()/.save() is marked for Databricks/Unity Catalog validation.

How the LLM Review Layer Works

The LLM is not asked to blindly read the repository. It is given a precise queue and a file-opening contract. This keeps the review focused, auditable, and cheaper.

Triage order
lineage_report.md→change_summary.md→llm_review_manifest.json→previous LLM outputs→lineage_edges.json→unresolved_items.json
Opening rule
For analyzed files, open review_queue[].content_store_path or map exact file_path through content_index.json. For inventoried files, open the raw repo path only when selected.
Incremental rule
If a file is unchanged and previous review exists, carry forward the previous decision. Do not reopen content just because old unresolved items still exist.
Status rule
probable = engine evidencecandidate = LLM-found static evidencellm_reviewed = LLM reviewed scanner edgeconfirmed = only after Databricks runtime / Unity Catalog validation

High priority

Unresolved items, analysis errors, parser failures, and unsafe partials. The LLM opens full normalized content and traces the logic manually.

Medium priority

Zero-edge files, low confidence, partial edges, temp-view chains, or unqualified table names. The LLM verifies whether lineage was missed or correctly absent.

Low priority

Unchanged files and workflow/config references. On incremental runs, these are mostly carry-forward unless the previous review is missing or incomplete.

Cache, Hashes, and Incremental Correctness

The engine separates raw file change detection from normalized content change detection. This is why metadata-only notebook changes do not force the LLM to reread irrelevant notebook JSON noise.

Cache key model

cache_key = sha256( engine_version + file_sha256 + config + sqlglot_available )
  • file_sha256 is the raw source file hash.
  • text_sha256 / content_sha256 is the normalized readable text hash.
  • Cache files are content-versioned; old cache files can remain without corrupting current output.

Metadata-only notebook change

Where savings come from

  • Scanner runtime: .cache/ skips notebook parsing, SQL parsing, AST traversal, content-store generation, and edge reconstruction for unchanged files.
  • LLM input tokens: change_summary.md and llm_review_manifest.json prevent reopening unchanged content.
  • LLM reasoning time: llm_reviewed_edges.json and llm_review_log.md become the carry-forward baseline.

Practical enterprise interpretation

For this repository size, LLM cost is already low. The more important enterprise benefit is deterministic repeatability: every review can explain which files changed, which edges were reused, which unresolved items were rejected, and which notebook still needs Data Engineering action.

Final system takeaway for technical stakeholders

This system should be positioned as an auditable static lineage accelerator. The scanner creates deterministic evidence. The LLM performs focused reasoning only where static parsing is weak. The second run is efficient because the engine cache avoids repeated parsing and the LLM carry-forward policy avoids repeated content review. The final enterprise control is runtime validation against Databricks Unity Catalog or another approved production lineage source.
